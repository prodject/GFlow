GControl coverage log
Updated: 2026-07-21

Source inventory
- .example: old Monji APK decompile and bundled resources.
- .src/Functions.txt: functional target list extracted from old app.
- .src/Ext: system APKs and decompiled sources used as integration references.

Key system sources found in .src/Ext
- OneOS-SystemUIPlugin_src: com.ecarx.xui.adaptapi.* including car, hvac, vehicle, dvr, evs, hudinteraction, tpms, wifiap, audiofx.
- ECarXCarService_src: ecarx car service and permission ecarx.car.permission.CAR_HVAC.
- CarService_src: Android Automotive car service permissions and vehicle categories.
- DIMInteraction_src: DIM/QNX interaction, navigation/phone/camera monitor classes.
- AudioExtService_src: autolink InteractionProto messages for dock/DIM/audio/media/navigation.
- OneOS-Dock_src and OneOS-ControlBoard_src: OneOS UI/cloud/navigation/dock integration classes.
- MultiWindowService_src and MultiWindowAnimator_src: privileged split/multi-window behavior.
- TCAMService_src: car vendor extension service entry points.

Coverage status legend
- implemented: functional code exists in GControl.
- partial: user-facing UI exists and some system/API calls are attempted.
- adapter: reflection-based system contract added; requires target firmware classes/permissions.
- scaffold: entry point/service exists but real behavior is not implemented.
- missing: not implemented yet.
- privileged: requires system signature, root, or adb grants.

12. Automation / smart logic
- Smart presets: implemented. Users can create app-defined command presets from AdaptAPI function/zone/value lines, including float HVAC temperature writes.
- Trigger engine: partial. Boot, foreground-app and manual trigger types are wired; deeper car-state triggers still need confirmed firmware signals.
- Steering wheel button mapping: partial. The receiver stores the last key event and can map keyCode + gesture (press/hold/double) to a smart preset when firmware broadcasts enough key metadata.
- Steering wheel advanced mapping: partial. Settings -> Steering buttons supports press/double/triple/hold gestures, held-button modifiers, behavior modes (replace/together/hold-only/stationary-only), target types (preset/scenario/action/voice/launch/command) and conditions (always/app/profile/temperature plus placeholders for stationary/moving until speed/gear signals are mapped).
- User profiles: partial. Profiles save/recall seat memory slots and optionally run a smart preset for climate/drive/light settings. Actual seat memory execution remains firmware-dependent.
- Driver/passenger profiles: partial. Separate driver and passenger profile lists can store/apply available GControl settings: seat memory and seat axes, mirror adjust mode, climate temperature/fan, seat heat/ventilation, drive mode, steering effort, HUD, brightness, ambience color, media volume, media source marker, desktop pinned apps, steering button mappings, custom presets/scenarios and ADAS toggles. Face ID/phone/Bluetooth/digital-key identifiers are stored as matching metadata; automatic detection depends on future system events.
- Smart climate controller: partial. SmartClimateController now runs as a mode-based controller (off, fast cooling, fast heating, stabilization, maintain temperature, HVAC drying, summer algorithm) instead of a fixed preset. It enforces a one-minute change cooldown, reads cabin/outside temperature and basic vehicle-state diagnostics through AdaptAPI ISensor when available, falls back to UI values when firmware signals are unavailable, controls driver/passenger zones separately and supports drying after trip. Call/fogging detection still needs separate phone/HVAC heuristic integration.
- Welcome/leave scenarios: partial. Template installer creates user-editable smart presets for profile recall, climate, ambience light, drive mode, window/sunroof close and lock actions, with manual triggers until confirmed door/approach/leave signals are mapped.
- Parking guard: partial. Template installer creates a smart preset that starts DVR, opens 360 through the custom key, closes windows/sunroof/sun curtain and locks doors. Gear/park-state trigger still needs confirmed vehicle signal mapping.
- Rain scenario: partial. Template installer creates a smart preset that closes all windows, sunroof and sun curtain, switches wipers to AUTO and enables front defrost. Automatic rain-sensor trigger still needs confirmed signal mapping.
- Night mode: partial. Template installer creates a smart preset for HUD navigation display, night daymode, low backlight/DIM brightness, blue DIM theme and subdued ambience lighting. Time/sunset trigger still needs scheduling or confirmed vehicle clock/day-night signal wiring.
- App context automation: partial. Template installer creates navigation app triggers that enable autozoom, HUD navigation display and stock navigation/DIM map custom-key actions. True split-window launch still needs privileged multi-window integration.
- Scenario engine v2: partial. User-editable scenarios support manual/app/boot/voice/steering-button triggers, conditions, repeat policies, cancel-on-condition-change, sequential steps, delays, wait conditions, notifications, app launch, voice/broadcast commands, smart-preset calls, actions and execution log with skip reasons. Many vehicle-state conditions remain diagnostic placeholders until engine/gear/speed/fuel/door/security/Bluetooth signals are mapped.
